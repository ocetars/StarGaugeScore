# 评分配置生产 skill 设计

## 1. 背景与目标

诉求起点是「让 SSS 对藿藿和遐蝶意味着差不多的投入水准」。拆解后得到三条结论，重塑了整个方向：

1. **跨角色评级等价靠 score.json 解决不了。** 权重决定的是「绝对毕业度」（离该角色理论满配多近）。角色满配难度不同是真实的绝对差异，调权重消不掉。
2. **不做跨角色矫正（不划曲线）。** 经数据核对，遐蝶与藿藿在评分器里结构几乎相同（都是 3 个满价值副词条、subMax 一致），算法**没有**内在惩罚奶妈。prod 里的分差主要是**玩家投入差异**（人群效应），是真实信息，不抹平。采用「绝对公平」取向：同等投入≈同等评级今天就已成立。
3. **因此唯一的杠杆是「让配置足够客观」。** 既然不矫正，每一分的全部重量都压在配置上、没有兜底。「客观」= 权重客观且尺子统一 + 模型物理诚实。

本规格交付的是承载这件事的工具：一个**能采集社区共识、内建评分尺子与框架、针对角色产出评分配置**的 skill。它把「一套可复核的判据」固化成可执行流程，让任何角色（新角色、或重做存量角色）都能按同一把尺子出分。

## 2. 已定结论（设计约束）

- **绝对评分语义**：分数恒为「离该角色完美遗器有多近」，跨角色平均分差异是真实信息。
- **不划曲线**：不引入任何「按角色人群分位」的评级矫正。评级阈值仍全局。
- **不做 breakpoint-aware 评分**：评分器是「只看遗器的纯函数」，拿不到光锥/行迹/星魂的整体面板，无法判断阈值词条是否过线。「过线归零」式评分明确 out-of-scope（YAGNI）。阈值词条压成单个保守权重。
- **客观 = 权重客观（尺子统一）+ 模型物理诚实**。本 skill 专注前者；模型物理诚实（主副比、subMax 次数模型、等级曲线、主词条单值限制）归入独立的「模型校准」工作，见 §9。

## 3. 交付形态

- **类型**：skill，置于 `.claude/skills/star-gauge-score-author/`（与 `star-gauge-version-update`、`star-gauge-db` 同级；工作区根的 skill 不纳入 git，符合既有惯例）。
- **结构**：
  - `SKILL.md`：触发条件、内建判据（§4 全文）、工作流（§6）、校验（§7）。
  - `references/profiles.md`：角色画像活登记（§4.3，随处理增长，初始可空）。
  - `references/stat-keys.md`：社区口径属性名 ↔ 人工表字段 ↔ score.json 内部键 的对照（§5）。
- **作用域**：按角色驱动，支持单个 / 一批 / 全量。「针对一个角色出分」是最小单元；「重做全部存量角色」只是对全表跑一遍。
- **调用**：用户说「给某角色制定评分 / 重做某角色权重 / 校一遍评分配置客观性」时触发；版本更新 skill 的「为新角色补词条价值」一步将改为引用本 skill。

## 4. 内建判据（skill 的核心内容）

### 4.1 五档语义锚

权重取固定五档 `{1, 0.75, 0.5, 0.25, 0}`。每档配一个**可判定问题**，填值时回答问题而非凭印象：

| 档 | 名 | 判定问题（满足即归此档） |
| --- | --- | --- |
| 1.0 | 核心 | 直接乘该角色主输出/主职能的杠杆，且「永远想要更多」（无软上限或上限极高）。去掉它核心数值线性下跌。 |
| 0.75 | 高优次选 | 仍乘输出杠杆但有边际递减，或是「第二条必备」。属 BiS，排在核心之后。 |
| 0.5 | 有用次要 | 贡献职能但非主线，或有明显软上限的辅助向收益。 |
| 0.25 | 边缘填充 | 仅在某阈值前有限有用，或微弱生存向；有比没有强一点，但不会专门去刷。 |
| 0 | 无用 | 不影响该角色职能，堆它零贡献。 |

### 4.2 阈值词条规则（速度 / 效果命中 / 效果抵抗 / 击破）

阈值词条只能压成单权重 = 典型边际价值，判定靠「光靠遗器副词条能否轻易冲过线」：

- **速度**：副词条单跳约 +2，难仅靠副词条大幅超线 → 线性近似安全 → 按角色需求给 **0.5～0.75**（按角色浮动，不钉死）。
- **效果命中**：能靠副词条堆很高、容易超线 → 线性会高估超线部分 → 给**保守档**（核心需求 0.75 / 够线即可 0.5），文档注明是近似。吃效果命中的角色（卡芙卡、黑天鹅、佩拉）身件主词条仍可 1.0，但**副词条**效果命中按 0.75 而非 1。
- **击破**：对击破角色越多越好、近似无软上限 → 当核心 **1.0**，不属阈值问题。
- **效果抵抗**：纯生存、价值低且无输出阈值 → 多为 **0.25**，无关角色给 **0**。

### 4.3 一致性原则与角色画像登记

评分一致性靠两样东西保证：

- **主力**：§4.1 判定问题 + §4.2 阈值规则**统一套用**。它们是客观规则，对所有角色一视同仁，本身就提供大部分一致性，不需要预定义分类。
- **防漂移**：**同画像角色对齐**，catch 两个结构相同的角色被独立研究出不同值（「暴击给 A 打 1、给 B 打 0.9」）的手抖。

**不预先定义画像目录**——玩法多样、随版本迭代，自造一套固定分类既不稳又僭越。画像一律按**社区对该角色的定位**认定；我们只保留「出分前要考虑角色画像、并让同画像角色对齐」这条意识。

机制（描述性、随处理增长，非预先穷举）：

1. 每个角色从社区归纳**画像** = 定位（输出/辅助/生存…）+ 加成主属性（攻击/生命/防御）+ 关键机制（暴击/击破/持续伤害/增伤/护盾…）。
2. 出分前查 `references/profiles.md`（活登记）有无**同画像先例**：
   - 有 → 对齐到先例档位，偏离须记录理由 + 来源。
   - 无 → 新画像，按 §4.1/§4.2 从社区定档，并把该画像 + 档位骨架追加进登记。
3. 登记从「已做角色 + 社区」长出来，自然覆盖新机制、新版本，无需我们预先发明分类。

> 加成主属性是独立维度：同为「暴击输出」机制，遐蝶（生命）与黄泉（攻击）画像不同（主属性%槽不同），各自对齐各自先例。

### 4.4 主词条规则

沿用同一套五档与判定问题，现实里更两极：BiS 主=1.0；有产出损失的可选主=0.5；基本错配=0.25；无用=0。速度鞋这类「人人都要」按 1.0，攻击鞋等可选按 0.5。头/手主词条固定无选择、不参与评分。

> 注意 generate 的归一化：每部位最高值会被拉到 1。所以**必须把真正的 BiS 主显式给到该部位的最高值**（通常 1），否则一个「全是低值」的部位会被归一化伪造出满分主词条。

### 4.5 映射规则（社区共识 → 档位）

- 社区给的是**有序优先级**（「暴击>暴伤>攻击%>速度」、「速度堆到 X」），不是数值。
- 按「这条在该角色 **BiS 里的地位**」对号入座到 §4.1 的判定问题，**不按名次机械切**——同样排第四，在暴击DPS可能 0.5、在别处可能 0.25。名次只是线索，判定问题才是尺子。
- 优先采用**当前游戏版本**的共识。

## 5. 数据管线对接

人工表用**社区口径属性名**，`generate.ts` 翻译成 score.json 的内部键。skill 写的是人工表，必须吃透 generate 的变换，否则产出会被二次变换走样。

- **两份人工表**（`data/mainAffixValues.json` / `data/subAffixValues.json`）：每角色一行，键为社区口径名。
  - 主表字段：`Attack` `HP` `Defence` `Speed` `CriticalChance` `CriticalDamage` `StatusProbability`(效果命中) `BreakDamage`(击破) `DamageAddedRatio`(球伤害加成) `SPRatio`(能量绳) `HealRatio`(治疗身，可选)。
  - 副表字段：`Attack` `HP` `Defence` `Speed` `CriticalChance` `CriticalDamage` `StatusProbability` `StatusResistance`(效果抵抗) `BreakDamage`。
- **generate 的关键变换**（设计与校验都要按它推理）：
  1. 主词条每部位最高值归一到 1（见 §4.4 注意）。
  2. 绝对值词条 flat = %值/3 自动推导，flat 不手填。
  3. subMax 自动 = 排除该部位最优主词条后 top4 副词条 × 乘数（头/手 `[6,1,1,1]`，身/脚/球/绳 `[5,1,1,1]`）。rubric 不直接设分母，但副词条质量决定它。
  4. 球攻击修正：高伤害+高攻击时球上攻击% 取 `max(伤害*0.8, 攻击)`。
- **已知模型限制**（写入文档，归 §9 而非本 skill 修）：主词条价值是**按属性单值**、非按 (部位,属性)。即 ATK% 在球和身共用一个值，无法表达「球要攻击%、身要暴击」式的细分；靠「每部位可选属性不同 + 归一化」缓解，但存在 ATK%/HP%/DEF% 在多部位重叠时的表达力损失。

## 6. skill 工作流

针对一个或一批角色：

1. **定位**：从 `data/raw/avatarConfig.json` 取 id / 命途 / 伤害属性（给线索），结合社区认定角色**画像**（定位 + 加成主属性 + 关键机制）。新角色先在 avatarConfig 补行。
2. **采集**：检索当前版本社区共识（Prydwen / game8 / 攻略 / 各评分器），取主词条与副词条优先级、阈值。记录来源。
3. **对画像**：查 `references/profiles.md` 活登记；有同画像先例则对齐其档位，无则按 §4.1/§4.2 定档并把新画像入登记。
4. **定档**：按 §4.1 判定问题与 §4.2 阈值规则给每个属性定档；偏离模板处记录理由 + 来源。
5. **写表**：把社区口径值写入 `mainAffixValues.json` / `subAffixValues.json`（保持 id 升序，diff 友好）。
6. **生成**：`npm run generate` → score.json（同时推导 subMax）。
7. **回归**：`npm test`（金样本：生成对齐 + 评分器固定向量）必须全绿。
8. **校验**：§7 四关。
9. **后端同步**（仅在用户要落地时）：复制 score.json 到 `StarGauge-server/src/assets/score.json` → 构建 → 部署 → 按需重算 prod。这一步与 `star-gauge-version-update` 的 §3 一致，引用之而非重复。

## 7. 校验四关

1. **同画像一致性**：同画像角色底表应高度相同；对照活登记比对、发散即报警（catch「暴击给A打1、给B打0.9」）。
2. **来源可溯**：每个非 0 权重要么有社区来源、要么标「继承自 X 模板」；无依据的不许凭空，标「待定」。
3. **人工抽检**：列出本批改动，请用户挑几个熟角色眼检。
4. **回归 + 重标定**：金样本固定向量不破；存量角色大改后用 `scripts/calibrate.ts` 重跑、按目标分位重定全局评级阈值。

## 8. 范围与非目标

**范围内**：建 skill（SKILL.md + 两个 reference）；判据全文固化；按角色出分的工作流与校验。

**非目标（YAGNI）**：
- 不做 breakpoint-aware 评分（架构所限）。
- 不做跨角色评级矫正/划曲线。
- 不在本 skill 内改 generate 的算法或数据模型（主副比、subMax 乘数、主词条单值限制等归 §9）。
- 不强制连续权重——固定五档是刻意选择，为一致性让渡粒度。
- 不预先穷举/自造画像分类——画像按社区认定、写进随处理增长的活登记（§4.3）。

**预期首个应用**（独立于「建 skill」，建完再单独决策投入）：用 skill 对全部存量角色跑一遍重导，修掉现状的尺子不一（0.1 兜底独大、0.88 野值等），再重标定阈值。

## 9. 后续：模型物理诚实（独立工作）

与本 skill 平行、不阻塞它的另一条客观性杠杆，待单独立项：

- 主副比 `0.35 / 0.65`：错主词条的件还能捞 0.65，惩罚是否够。
- subMax「顶级副词条配 5 次强化」是否贴合游戏真实强化上限。
- 等级曲线 `(lvl+1)/16` 是否合理。
- 主词条价值「按属性单值」的表达力限制（§5）是否值得升级为按 (部位,属性)。

## 10. 风险与开放项

- **社区数据时效/幻觉**：靠「必须引源 + 认版本 + 校验四关的对抗检查」兜底；拿不准标「待定」而非瞎给。
- **新/冷门角色无共识**：标「待定」，不硬编；有同画像先例可对齐则对齐。
- **画像边界模糊**（如混合定位角色）：执行时归主定位 + 记录偏离，作为一条新画像入登记。
- **活登记冷启动**：初期登记为空、前几个角色无先例可对齐，全靠 §4.1/§4.2 判定。这是预期的基况——判定问题与阈值规则本就是一致性主力，登记只是防后续漂移。

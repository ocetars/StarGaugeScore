# 后续考虑（Backlog）

v1 只做「同原材料、更合理的合成算法」。以下是讨论中明确但本期不做的事，按价值排序。

## 1. 边际价值函数（角色化非线性定价）

当前副词条按 `count × 固定权重` 线性计分。但属性价值是非线性的：

- 暴击/暴伤有最佳配比（约 1:2），堆偏了边际收益骤降；
- 速度卡阈值（如 134/160），是阶梯不是斜坡；
- 单属性堆过头收益递减。

**已验证的痛点**：用真实数据对比两个遐蝶毕业号——线性算法把双暴更弱的号排在更强的号前面（排序算反）。
非线性边际价值能纠正，但需逐角色定制价值曲线，**精准但不普适**，工程与维护重。
适合在玩家普遍高练度、需要在卷王之间拉开差距时再做。

## 2. IMPOSSIBLE 档（数据驱动、有流动性）

最高档不用固定阈值，而取该角色玩家实时分布的 top 1%，随数据流动。
需要：足够样本量门槛、新角色冷启动兜底。v1 评级体系暂不含此档。

## 3. 评级阈值标定

v1 合成算法整体下移分数，需重新定一套全局固定阈值（B…GODLIKE）。
**不追求各角色评级比例一致**——已确认：难养成的角色分数本就偏低，这是真实信息，不应被算法抹平。
标定只做一次、全局通用。

## 4. 与后端的算法同步机制

v1 靠「金样本对齐」让后端实现与本仓库参考实现一致（人工保证）。
后续可考虑：把金样本向量文件化、后端 CI 直接消费本仓库导出的向量做回归，杜绝两边漂移。

## 5. 自维护权重表的质量保障

`data/mainAffixValues.json` / `subAffixValues.json` 是人工判断资产。可加：
- 新角色权重的录入校验（属性 key 合法、值域 0~1）；
- 与社区/官方推荐的交叉核对流程。
